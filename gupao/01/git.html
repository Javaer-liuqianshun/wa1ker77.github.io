# git

version control（版本控制）

# git和svn的区别

  1. svn是集中式版本控制；git是分布式版本控制
  2. svn一次push是存储的变换；git一次push存储的是完整内容，如果没有变换存储的是一个指针

集中式版本控制概念：

版本库是集中放在中央服务器的，而干活的时候，用的都是自己的电脑，所以要先从中央服务器取得最新的版本，然后开始干活，干完活了，再把自己的活推送给中央服务器。

图1：集中式版本控制

集中式版本控制最大的缺点：必须联网才能使用。

分布式版本控制概念：

首先，分布式版本控制没有“中央服务器”，每个人的电脑上都是一个完整的版本库。当工作时，就不需要联网，因为版本库就在自己的电脑上。既然每个人都有一个完整的版本库，多人如何协作呢？比方说你在自己电脑上改了文件A，你的同事也在他的电脑上改了文件A，这时，你们俩之间只需把各自的修改推送给对方，就可以互相看到对方的修改了 。

在实际使用分布式版本控制时候，其实很少在两个人之间的电脑上推送版本库的修改，因为可能两个人不在同一局域网，两台电脑互相访问不了。因此，分布式版本控制通常也有一台“中央服务器”，但这个服务器的作用仅仅是用来方便“交换”大家的修改，没有它大家也一样干活。

图2：分布式版本控制

# git设置

设置和修改用户名和邮箱：

// git config命令的--global参数，用了这个参数，表示你这台机器上所有的Git仓库都会使用这个配置，当然也可以对某个仓库指定不同的用户名和Email地址

# 工作区和暂存区

工作区：

就是在电脑里能看到的目录

版本库（其中包含暂存区）：

工作区有一个隐藏目录.git，这个其实不算工作区，而是git的版本库。git版本库里面存储了很多东西，其中最重要的就是称为stage（或者叫index）的暂存区，还有git为我们自动创建的第一个分支master，以及指向master的一个指针叫HEAD。

# git常用命令

  1. git status查看工作区代码相对于暂存区的差别

  2. git add把修改的代码从工作区添加暂存区，如git add README.md；git add .，"."代表当前目录

  3. git commit把暂存区内容提交到本地版本仓库（当前分支），如git commit -m 'first commit',"-m"后面的表示本次提交的说明，也可不写，直接使用“git commit”

  4. git push origin master将本地版本仓库推送到远程服务器，origin是远程主机，master表示是远程服务器上的master分支，分支名是可以修改的

  5. git diff查看工作区代码和本地版本仓库代码有什么不同（即修改内容），如git diff README.md

  6. git log查看从最近到最远的提交日志，如git log --pretty=oneline，其中“--pretty=oneline”参数表示一行输出；如git log --graph --pretty=oneline --abbrev-commit命令可以查看到分支合并图

  7. git reset版本回退，如想回退到上一个版本使用git reset --hard HEAD^，回退到上上一个版本使用git reset --hard HEAD^^，回退到指定版本使用git reset --hard 版本号

  8. git reflog查看记录的每一次命令

  9. git checkout <bransh> --file恢复文件，其实是用本地版本库替换工作区的版本，无论工作去是修改还是删除，都可以"一键还原"；git checkout <branch>切换分支

  10. 撤销修改：

    1. 当改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout --file
    2. 当不但改乱了工作区某个文件的内容，还添加到了暂存区，想丢弃修改，分两步，第一步用命令git reset HEAD <file>，此时和1一样，按1操作
    3. 已经提交了不合适的修改到版本库，先撤销本次提交，使用git reset --hard HEAD^回退版本

  11. git rm删除文件，如git rm README.md删除文件，并使用git commit -m 'remove README.md'来提交删除文件操作

  12. git branch查看当前分支，当前分支前面会加“*”

  13. git merge <branch name>合并分支，如在master分支中合并dev分支：git merge dev

  14. git branch -d <branch name>删除分支，如删除dev分支git branch -d dev，强制删除分支使用git branch -D <branch name>

  15. git remote查看远程仓库名称，git remote -v查看具有远程仓库的权限

# 远程仓库

使用 git remote add origin git@github.com:wa1ker77/learngit.git 把本地仓库和远程仓库建起连接。

将本地仓库的所有内容推送到远程仓库git push -u origin master，由于远程仓库是空的，第一次推送master分支时，加上-u参数，git不但会把本地的master分支内容推送到远程的master分支，还会把本地的master分支和远程的master分支关联起来，以后推送或拉取时可以简化命令。

使用git clone git@github.com:wa1ker77/learngit.git把远程仓库代码克隆到本地仓库。

使用git push origin <branch name>推送本地版本库到远程版本库，如git push origin master。

在本地创建和远程分支对应的分支 ，使用git checkout -b branch-name origin/branch-name，本地和远程分支的名称最好一致 。

如果git pull提示no tracking information，说明本地dev分支与远程origin/dev分支没有链接，需要设置本地dev分支和origin/dev分支的链接：git branch --set-upstream-toset dev origin/dev。使用git pull如果有冲突先处理冲突。

# 分支管理

创建与合并分支:

在git里，一开始只有master分支，master分支是一条线，git用master指向最新的提交，再用HEAD指向master，就能确定当前分支，如图：

图4：master和HEAD说明

每次提交，master分支都会向前移动一步，这样，随着你不断提交，master分支线也越来越长。

git chekout -b dev创建dev分支，并切换到dev分支。git checkout命令加上-b参数表示创建并切换，相当于git branch dev;git checkout dev两条命令。

修改README.md文件，使用git add README.md;git commit -m 'git study for branch在dev分支里面提交README.md到本地版本库。再使用git checkout master切换到master分支，最后使用git merge dev来合并dev分支。

解决冲突：

创建一个feature1分支，在feature1分支中修改README.md，使用git add README.md;git commit -m 'git study by feature1'。然后切换到master分支，在master分支中修改README.md，使用git add README.md;git commit -m 'git study by master'。现在master分支和feature1分支都有新的提交，变成如图所示：

在master分支使用git merge feature1git会提示README.md文件存在冲突，必须手动解决冲突后再提交。修改后使用git add README.md;git commit -m 'conflict fixed'，此时master分支和feature1分支变成如图所示：

注意：git合并分支时，使用Fast forward模式，这种模式下，删除分支，会丢掉分支信息。使用--no-ff方式禁止使用Fast forward模式。如git merge --no--ff -m 'merge with no--ff' feature1。因为本次合并要创建一个新的commit，所以加上-m参数。

# 分支bug

在git中，每个bug都可以通过一个新的临时分支来修复。修复后，合并分支，然后将临时分支删除。

当在dev分支下进行开发时，master分支出现bug，需要立即修改。但是此时因为dev并没有开发完，还不能提交，并bug需要立即更改，就需要git提供的stash功能，可以把当前工作现场“储藏”起来，等以后恢复工作现场后继续工作。

在dev分支下使用git stash把dev分支“储藏”，git checkout master切换到master分支处理bug，并为***代号为101的bug***新开一个分支进行处理git checkout -b issue-101，修改完bug后在master分支下合并issue-101分支git merge --no--ff -m 'merged bug fix 101'并删除issue-101分支git branch -d issue-101。然后切换git checkout dev到dev分支下，使用git stash list查看“储藏”的工作现场，使用git stash apply或git stash pop来恢复工作现场，git stash apply恢复后stash内容并不删除，还需使用git stash drop来删除，而git stash pop 恢复的同时，删除stash内容。

# 分支标签

  1. git tag <tagname>用于创建一个新标签，默认HEAD，也可以指定一个commit id

  2. git tag -a <tagname> -m "remark"可以指定标签信息

  3. git tag查看所有标签

  4. git push origin <tagname>可以推送一个本地标签；

  5. git push origin --tags可以推送全部未推送过的本地标签；

  6. git tag -d <tagname>可以删除一个本地标签；

  7. git push origin :refs/tags/<tagname>可以删除一个远程标签。
