# 线程安全性

编写线程安全的代码，本质上就是管理对状态的访问，而且通常都是共享的、可变的状态。“共享”意味着变量可以由多个线程同时访问，“可变”意味着变量的值在其生命周期内可以发生变化。

Java中主要的同步机制是关键字sychronized，它提供了一种独占的加锁方式，但“同步”这个术语还包括volatile类型的变量，显示锁(Lock)以及原子变量。

无状态对象一定是线程安全的。

# 原子性

原子性指该操作是不可再分的，该操作要么全部执行，要么一点都没有执行。

竞争状态的“先检查后执行”和“读取 - 修改 - 写入”操作其实是复合操作，为了保证线程安全，操作必须原子地执行。

@ThreadSafe
public class CountingFactorizer implements Servlet{
    // 计数器使用原子变量类来管理
    private final AtomicLong long count = new AtomicLong(0);
    
    public long getCount(){
        return count.get();
    }
    
    public void service(ServletRequest req,ServletResponse resp){
        BigInteger i = extractFromRequst(req);
        BigInteger[] factors = factor(i);
        count.incrementAndGet();
        encodeIntoResponse(resp,factors);
    }
}

“读取 - 修改 - 写入”复合操作使用线程安全类 - 原子变量类管理计数器的状态，保证线程安全性。当只向无状态类中加入唯一的状态元素，而这个状态被线程安全的对象所管理，那么新的类仍然是线程安全的。

# 锁

如果想向无状态类加入更多的状态，使用原子变量类无法保证线程安全性。

@NotThreadSafe
public class UnsafeCachingFactorizer implements Servlet{
    private final AtomicReference<BigInteger> lastNumber = new AtomicReference<BigInteger>();
    private final AtomicReference<BigInteger[]> lastFactors = new AtomicReference<BigInteger[]>();
    
    public void service(ServletRequest req,ServletResponse resp){
        BigInteger i = extractFromRequst(req);
        if(i.equals(lastNumber.get())){
           	encodeIntoResponse(resp,lastFactors.get());
        }else{
           	BigInteger[] factors = factor(i);
            lastNumber.set(i);
            lastFactors.set(factors);
            encodeIntoResponse(resp,factors);
        }
    }
}

UnsafeCachingFactorizer有两个共享状态，即使两个共享状态都是原子引用，但是还是可能破坏不变约束。UnsafeCachingFactorizer不变约束是：缓存在lastFactors中的各个因子的成绩应该等于缓存在lastNumber中的数值，所以UnsafeCachingFactorizer是线程不安全类。

为了保护状态的一致性，要在单一的原子操作中更新相互关联的状态变量。单一的原子操作使用Java提供的强制原子性的内置锁机制synchroized块来保证。

# 线程重入

当某个线程请求一个由其他线程持有的锁时，发出请求的线程就会阻塞。然而，由于内置锁时可重入的，因此如果某个线程试图获得一个已经由它自己持有的锁，那么这个请求就会成功。“重入”意味着获取锁的操作的粒度是“线程”，而不是“调用”。

重入的一种实现方法是，为每个锁关联一个获取计数器和一个所有者线程。当计数器为0时，这个锁就被认为是没有被任何线程持有。当线程请求一个未被持有的锁时，JVM将记下锁的持有者，并且将获取计数器置为1。如果同一个线程再次获取这个锁，计数器将递增，而当线程退出同步代码块时，计数器会相应地递减。当计数器为0时，这个锁将被释放。
